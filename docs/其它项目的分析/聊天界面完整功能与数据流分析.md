## 二、用户消息发送

### 2.1 消息组成

| 组成部分 | 来源 | 处理方式 | 限制 |
|---|---|---|---|
| **文本** | 编辑器输入 | 可选添加努力前缀（`[thinking]...[/thinking]`），替换终端上下文占位符 | ≤120,000 字符 |
| **图片** | 文件选择/拖拽 | `File` → `FileReader.readAsDataURL()` → Base64 Data URL | 最多 8 张，每张 ≤10MB，Data URL ≤14,000,000 字符 |
| **终端上下文** | 终端面板 | 替换占位符为终端输出内容，以代码块追加到文本末尾 | — |

### 2.2 发送流程

```
用户输入文本/选择图片 → ChatComposer 组件
  ↓ 点击发送 / Ctrl+Enter
performChatViewSend()
  ├─ 1. 获取编辑器的 sendContext()
  │   { prompt, images, terminalContexts, modelSelection, runtimeMode, interactionMode }
  ├─ 2. deriveComposerSendState() 验证是否有可发送内容
  ├─ 3. appendTerminalContextsToPrompt() 替换终端上下文占位符
  ├─ 4. formatOutgoingPrompt() 根据模型添加努力前缀
  ├─ 5. 图片遍历：readFileAsDataUrl() 转 Base64
  ├─ 6. 乐观更新：先将用户消息插入本地 store
  │   { messageId, role: "user", text, attachments, streaming: false }
  ├─ 7. RPC 调用：api.orchestration.dispatchCommand()
  └─ 8. 失败处理：恢复草稿，移除乐观消息
```

### 2.3 RPC 调用格式

```json
{
  "method": "orchestration.dispatchCommand",
  "params": {
    "type": "thread.turn.start",
    "commandId": "c-uuid",
    "threadId": "t-uuid",
    "message": {
      "messageId": "m-uuid",
      "role": "user",
      "text": "格式化后的提示文本（含努力前缀 + 终端上下文）",
      "attachments": [
        {
          "type": "image",
          "name": "filename.png",
          "mimeType": "image/png",
          "sizeBytes": 12345,
          "dataUrl": "data:image/png;base64,iVBOR..."
        }
      ]
    },
    "modelSelection": {
      "provider": "claude",
      "model": "claude-sonnet-4-6"
    },
    "titleSeed": "对话标题的种子文本（截断后）",
    "runtimeMode": "full-access|auto-accept-edits|approval-required",
    "interactionMode": "default|plan",
    "createdAt": "2026-05-08T00:00:00.000Z"
  }
}
```

## 三、服务端事件系统

### 3.1 核心数据流

```
用户消息到达服务端 ws.ts
  ↓
OrchestrationEngine.dispatch()  →  decider 处理
  ├─ thread.message-sent（持久化用户消息）
  └─ thread.turn-start-requested
      ↓
ProviderCommandReactor.processTurnStartRequested()
  ├─ ensureSessionForThread()     ← 复用或创建会话
  └─ providerService.sendTurn()
      ↓
ClaudeAdapter.sendTurn()
  ├─ 构建 Claude SDK 的 SDKUserMessage
  ├─ 推入 promptQueue
  └─ SDK AsyncIterable 消费消息 → claudecode 处理
      ↓
handleSdkMessage() 循环处理 SDK 事件
  ├─ stream_event  → 流式内容/工具调用/计划更新
  ├─ system       → 任务进度/状态变化/文件持久化
  ├─ user         → 工具执行结果（tool_result）
  ├─ assistant    → 后台代理响应
  ├─ result       → turn 完成
  └─ telemetry    → 工具进度/速率限制/使用摘要
      ↓
ProviderRuntimeEvent → ProviderRuntimeIngestion
  ↓
InternalOrchestrationCommand → 持久化 + 事件流
  ↓
WebSocket 推送到客户端 (subscribeThread)
```

### 3.3 服务端 → 客户端推送协议

客户端通过两个 WebSocket RPC 方法订阅数据流：

```
orchestration.subscribeThread(threadId)
  → 返回 AsyncIterable<OrchestrationThreadStreamItem>
```

流项目有三种类型：

```json
// 1. 初始快照（订阅时一次性发送）
{
  "kind": "snapshot",
  "snapshot": {
    "snapshotSequence": 1,
    "thread": {
      "id": "thread-uuid",
      "messages": [ ... ],        // 所有历史消息
      "activities": [ ... ],      // 所有历史活动
      "proposedPlans": [ ... ],   // 所有计划提案
      "sessions": { ... },        // 当前会话
      "turnDiffs": [ ... ]        // turn diff 摘要
    }
  }
}

// 2. 实时事件（持续推送）
{
  "kind": "event",
  "event": {
    "type": "thread.message-sent",
    "payload": { ... }
  }
}

// 3. 心跳（每 30 秒）
{
  "kind": "heartbeat",
  "timestamp": "2026-05-08T00:00:00.000Z"
}
```

推送的事件类型仅限于以下 6 种（只推送给关心该线程详情的客户端）：

| 事件类型 | 说明 |
|---|---|
| `thread.message-sent` | 消息新增/追加 |
| `thread.activity-appended` | 活动新增 |
| `thread.proposed-plan-upserted` | 计划提案创建/更新 |
| `thread.turn-diff-completed` | turn diff 生成完成 |
| `thread.reverted` | 已撤回 |
| `thread.session-set` | 会话状态更新 |

## 四、流式响应处理

### 4.1 核心机制：增量追加模式

**不是按 chunk 传输原始 token**，而是通过 `thread.message-sent` 事件携带增量文本：

```json
// 流式进行中（每次推送一个增量）
{
  "type": "thread.message-sent",
  "payload": {
    "threadId": "t-uuid",
    "messageId": "m-uuid",
    "role": "assistant",
    "text": "这是新生成的增量文本",
    "streaming": true,
    "createdAt": "2026-05-08T00:00:00.000Z",
    "updatedAt": "2026-05-08T00:00:01.000Z"
  }
}

// 流式完成后（最终推送）
{
  "type": "thread.message-sent",
  "payload": {
    "threadId": "t-uuid",
    "messageId": "m-uuid",
    "role": "assistant",
    "text": "这是完整的最终文本（取代所有增量）",
    "streaming": false,
    "createdAt": "2026-05-08T00:00:00.000Z",
    "updatedAt": "2026-05-08T00:00:30.000Z"
  }
}
```

### 4.2 客户端处理规则

```typescript
// 伪代码：流式消息处理逻辑
function handleMessageSent(thread, message) {
  const existingMessage = thread.messages.find(m => m.id === message.id);
  
  if (!existingMessage) {
    // 首次收到：创建新消息
    thread.messages.push(message);
  } else if (message.streaming) {
    // 流式中：追加文本
    existingMessage.text += message.text;
    existingMessage.updatedAt = message.updatedAt;
  } else {
    // 流式结束：替换为完整文本
    existingMessage.text = message.text;
    existingMessage.streaming = false;
    existingMessage.completedAt = message.updatedAt;
  }
}
```

### 4.3 流式渲染优化

| 优化点 | 做法 |
|---|---|
| 代码块语法高亮 | 流式中跳过缓存，使用 `isStreaming` 标志控制 |
| 复制按钮 | 流式过程中隐藏 |
| 实时计时器 | 自更新组件（1 秒间隔），显示 "Working for Xs" |
| 时间线合并 | `deriveTimelineEntries()` 按时间排序合并消息、活动、计划 |

### 4.4 思考过程（Reasoning）的显示

Claude Code SDK 发出的 `content_block_delta` 中 `delta.type: "thinking_delta"` 被转为 `content.delta` 事件（`streamKind: "reasoning_text"`），在 ingestion 层被转为 `task.progress` 活动。在 UI 上以工作日志条目的方式显示（`tone: "thinking"`），带有 `BotIcon` 图标和柔和色。

## 六、多轮对话生命周期

### 6.1 核心概念

T3code 使用 **"一个线程 = 一个 Claude Code 进程"** 的模型：

- **Session**：对应一个 Claude Code 进程（`claudecode` 二进制），通过 SDK 的 `query()` 创建
- **Turn**：对应一次"用户发送消息 → AI 完成回复"的完整交互
- **Thread**：对应 UI 中的一个聊天线程，包含多个 turn

同一个 thread 内，**session 保持活跃**，后续 turn **复用一个进程**。

### 6.2 会话创建（startSession）

```typescript
// ClaudeAdapter.ts 伪代码
async function startSession(input) {
  // 生成新 sessionId
  const sessionId = generateUUID();
  
  // 构建 SDK query 选项
  const queryOptions = {
    cwd: input.cwd,                                    // 工作目录
    model: apiModelId,                                  // 模型 ID
    pathToClaudeCodeExecutable: claudeBinaryPath,       // claudecode 路径
    sessionId: sessionId,                               // 新会话 ID
    resume: existingResumeSessionId,                    // 恢复用（后续轮次）
    permissionMode: permMode,                           // 权限模式
    allowDangerouslySkipPermissions: isFullAccess,      // 完全访问
    additionalInstructions: zhInstructions,             // 中文补充指令
    effort: effectiveEffort,                            // 努力级别
    canUseTool: permissionCallback,                     // 工具审批回调
    env: { ...env, ...process.env },                    // 环境变量
    settings: { alwaysThinkingEnabled, fastMode },      // 设置
    includePartialMessages: true,                       // 包含部分消息
  };
  
  // 调用 SDK query() → 内部 spawn claudecode 进程
  const queryRuntime = query({ prompt: promptQueue, options: queryOptions });
  
  // 启动异步流消费循环
  runSdkStream(queryRuntime);
  
  return { sessionId, resumeSessionId };
}
```

### 6.3 发送消息（sendTurn）

```
sendTurn()
  ├─ 检查是否有未完成的 turn，有则自动 complete
  ├─ 创建新 ClaudeTurnState
  ├─ 发出 turn.started 事件
  ├─ 构建 SDKUserMessage
  └─ 推入 promptQueue
      ↓
SDK AsyncIterable 消费到新消息
  ↓
claudecode 进程开始处理（同一进程！）
  ↓
handleSdkMessage() 循环处理流事件
  ↓
completeTurn() → 发出 turn.completed
```

```typescript
// ClaudeAdapter.ts 伪代码
async function sendTurn(input) {
  // 1. 自动完成上一个未完成的 turn（后台代理响应的情况）
  if (context.turnState && !context.turnState.completed) {
    await completeTurn(context, { interrupted: true });
  }
  
  // 2. 创建新 turn
  const turnState = {
    id: generateUUID(),
    items: [],
    startedAt: now(),
  };
  context.turnState = turnState;
  emitRuntimeEvent({ type: "turn.started", turnId: turnState.id });
  
  // 3. 构建 SDK 用户消息
  const sdkMessage = buildSdkUserMessage(input.message);
  
  // 4. 入队 → SDK 流会消费它
  await promptQueue.push(sdkMessage);
}
```

### 6.4 会话复用机制

```
Thread 中的第一个 turn：
  → 无活跃 session
  → startSession()  → 创建 claudecode 进程
  → sendTurn()      → SDK 处理

Thread 中的第二个 turn：
  → 已有活跃 session
  → 检查 runtimeMode / cwd / model 是否变化
  → 若不变：直接 sendTurn() → 复用一个进程
  → 若有变化：stopSession() → startSession() → sendTurn()
```

**状态变化会触发重建会话**（`ProviderCommandReactor.ts`）：

```
conditions:
  - runtimeMode 改变
  - cwd 改变（切换项目）
  - model 改变（切换模型）
    
action: stopSession() + startSession(newParams)
```

### 6.5 会话恢复（跨进程重启）

```
服务器重启后恢复会话：
  ┌─ 从 SQLite 读取 ProviderSessionRuntimeRepository
  │   → 找到 threadId 对应的 resumeCursor
  │   → { resume: "session-uuid", resumeSessionAt: "msg-uuid", turnCount: 5 }
  │
  ├─ adapter.hasSession()？ → 无（进程已结束）
  │
  └─ adapter.startSession({ resumeCursor })
      → query({ resume: "session-uuid", sessionId: "new-uuid" })
      → SDK 内部尝试恢复 claudecode 进程状态
      → 若成功：claudecode 继续之前的对话
      → 若失败：从头开始新会话
```

### 6.6 会话停止

```
stopSession()
  ├─ context.stopped = true
  ├─ 取消所有待处理审批
  ├─ completeTurn({ status: "interrupted" })
  ├─ 关闭 promptQueue
  ├─ 中断 SDK 流 fiber
  ├─ context.query.close()    → 终止 claudecode 进程
  ├─ 发出 session.exited 事件
  └─ 从 sessions map 删除
```

**停止后的会话不可恢复**，后续 `sendTurn` 会收到 `ProviderAdapterSessionClosedError`，需要重新 `startSession`。

## 十、WPF 复刻实现要点

### 10.1 通信协议

- 使用 WebSocket 连接（而不是 HTTP 轮询）
- 需要实现两个 RPC 方法：
  - `orchestration.dispatchCommand` → 发送命令（起 turn、切换模式）
  - `orchestration.subscribeThread` → 订阅线程事件流
- 协议是基于 JSON 的 Effect RPC（需实现请求/响应匹配）

### 10.2 需要处理的事件类型

| 优先级 | 事件 | 说明 |
|---|---|---|
| P0 | `thread.message-sent` | 流式消息处理（增量追加模式） |
| P0 | `thread.activity-appended` | 工具调用/思考/错误活动 |
| P0 | `thread.session-set` | 会话状态变化（驱动 Working 指示器） |
| P1 | `thread.turn-diff-completed` | 变更文件摘要 |
| P1 | `thread.proposed-plan-upserted` | 计划提案 |
| P1 | `thread.token-usage.updated` | Token 用量进度条 |
| P2 | `request.opened/resolved` | 审批流程 |
| P2 | `user-input.requested/resolved` | AI 提问 |
| P2 | `turn.plan.updated` | 任务清单 |

### 10.3 流式文本处理

- 收到 `streaming: true` 的消息 → 追加文本
- 收到 `streaming: false` 的同 ID 消息 → 替换为完整文本
- 代码块在流式过程中**不要**做高亮缓存

### 10.4 时间线排序

- 消息（`ChatMessage`）和活动（`OrchestrationThreadActivity`）分别存放
- 每次更新时按 `createdAt` 合并排序
- 工具调用日志需要合并连续的 `tool.updated` + `tool.completed` 为单条

### 10.5 会话管理

- 同一个 thread 复用一个 claudecode 进程
- 切换模型/模式/项目时需要重建会话
- 会话停止后不可恢复，需重新创建
